[Heading 1] 1. Executive Summary
[Normal] This hackathon challenges participants to build an Agentic AI Project Framework ù a multi-agent system that accepts a natural-language project specification and autonomously drives the full software development lifecycle, from requirements clarification through to a working, testable codebase. The challenge is not about building an AI model; it is about systems design, agent orchestration, failure recovery, and engineering rigor.
[Heading 1] 2. Background & Motivation
[Heading 2] 2.1  The Current State of AI-Assisted Development
[Normal] Modern AI coding assistants have dramatically accelerated individual developer productivity. Developers can describe a task in natural language and receive syntactically correct, contextually aware code within seconds. However, these tools share a fundamental limitation: they are reactive. A developer must manually break down a project, hand each piece to the AI, evaluate each output, and integrate the results. This model does not scale to teams, complex requirements, or long-horizon tasks that span multiple subsystems.
[Heading 2] 2.2  The Gap This Framework Fills
[Normal] What is missing is a proactive orchestration layer ù a framework that can:
[List Paragraph] Interpret a high-level, ambiguous project specification
[List Paragraph] Ask targeted clarifying questions to eliminate ambiguity
[List Paragraph] Decompose the specification into ordered, atomic implementation tasks
[List Paragraph] Delegate each task to an appropriate AI agent with the right tools
[List Paragraph] Validate outputs at every step before proceeding
[List Paragraph] Recover intelligently from failures without abandoning the entire workflow
[List Paragraph] Escalate to humans when ù and only when ù human judgment is genuinely required
[Normal] Solving this problem requires integrating ideas from multi-agent systems, software verification, human-computer interaction, and developer tooling ù making it an ideal deep-engineering hackathon challenge.
[Heading 1] 3. Problem Statement
[Normal] Design and implement an Agentic AI Framework that accepts a natural-language project specification and autonomously builds a working software project from scratch ù coordinating specialized AI agents to clarify requirements, architect a solution, generate and test code, audit for security, and produce a fully Dockerized, verified deliverable ù with minimal human intervention.
[Normal] This is a project hackathon, not a feature-addition hackathon. For security and neutrality reasons, no shared source code repository is provided as a base. Each team builds their framework and all generated projects entirely from scratch. The framework must demonstrate domain-agnostic capability across multiple complexity tiers.
[Heading 1] 4. Scope
[Heading 2] 4.1  In Scope
[List Paragraph] A natural-language intake interface (CLI prompt, web form, or chat UI) for project specifications
[List Paragraph] An automated requirement refinement module that identifies ambiguities and asks targeted clarifying questions
[List Paragraph] A codebase planning and architecture module that designs the project structure from scratch
[List Paragraph] A planning module that decomposes the specification into ordered, atomic implementation tasks
[List Paragraph] A code generation module that invokes any LLM of the team's choice using their own API keys
[List Paragraph] A validation module that runs generated test suites, linters, and type checkers after each step
[List Paragraph] A failure recovery module that retries, self-corrects, or escalates based on the type of failure
[List Paragraph] A configurable guardrails system (e.g., require human approval above a file-change threshold, flag security-sensitive patterns)
[List Paragraph] Dockerization of the generated project (optional but recommended ù see Section 9.1)
[Heading 2] 4.2  Out of Scope
[List Paragraph] Training or fine-tuning any AI model
[List Paragraph] Building a CI/CD pipeline or deployment system
[List Paragraph] IDE plugin development (terminal or web interface is sufficient)
[List Paragraph] Production-grade security hardening of the framework itself
[List Paragraph] Modifying or extending any pre-existing shared codebase ù all project code must be generated from scratch
[Heading 1] 5. Functional Requirements
[Normal] The following requirements are grouped by component. Requirements marked [CORE] are mandatory for a passing submission. Requirements marked [EXTENDED] are optional and will be rewarded in scoring.
[Heading 2] 5.1  Requirement Intake & Refinement
[List Paragraph] FR-01: [CORE]
[Normal] The framework must accept a natural-language project specification as its primary input and automatically identify underspecified aspects, generating a minimal set of targeted clarifying questions.
[List Paragraph] FR-02: [CORE]
[Normal] After receiving clarification responses, the framework must produce a structured specification document including: project summary, acceptance criteria, proposed architecture, and known constraints.
[Heading 2] 5.2  Architecture & Planning
[List Paragraph] FR-04: [CORE]
[Normal] The framework must design the project structure from scratch, determining the directory layout, modules, data models, and API contracts before any code is written.
[List Paragraph] FR-05: [CORE]
[Normal] The framework must decompose the project into an ordered list of atomic implementation tasks, where each task produces a verifiable, independently testable unit of work.
[List Paragraph] FR-06: [CORE]
[Normal] The implementation plan must be presented to the user for review and approval before execution begins.
[Heading 2] 5.3  Code Generation
[List Paragraph] FR-08: [CORE]
[Normal] The framework must invoke the team's chosen LLM via its API to generate code for each task, providing it with the structured specification, current project state, and any project-specific coding standards.
[List Paragraph] FR-09: [CORE]
[Normal] Generated code changes must be presented as a diff or summary to the user before being applied to the filesystem.
[Heading 2] 5.4  Test-Driven Validation
[List Paragraph] FR-11: [CORE]
[Normal] The framework must follow a TDD-first approach: the QA agent must generate failing test cases before any production code is written for each task.
[List Paragraph] FR-12: [CORE]
[Normal] After code generation, the framework must automatically run the test suite and treat any test failure as a blocking event that triggers the failure recovery module.
[List Paragraph] FR-14: [EXTENDED]
[Normal] The framework should run an AI-powered security audit pass after each major module is complete, flagging injection vulnerabilities, auth flaws, and logic errors.
[Heading 2] 5.5  Failure Recovery
[List Paragraph] FR-15: [CORE]
[Normal] When validation fails, the framework must automatically retry by passing the error output back to the code generation module, up to a configurable maximum number of retries.
[List Paragraph] FR-17: [EXTENDED]
[Normal] The framework should support rollback to the last passing checkpoint if the user chooses to abort mid-workflow.
[Heading 1] 6. Non-Functional Requirements
[Heading 1] 7. Project Complexity Tiers
[Normal] To ensure your engine is truly domain-agnostic and robust, the specific tasks will be withheld until the final phase. Your tool must generate code for 5 levels of complexity without any manual human code-level intervention.
[Heading 1] 8. Expected Output
[Normal] A complete, successful run of the framework must produce all of the following artifacts:
[Heading 2] 8.1  Intermediate Artifacts
[List Paragraph] Structured Specification Document ù A machine-readable (JSON or YAML) and human-readable summary of the refined project requirements, acceptance criteria, and proposed architecture.
[List Paragraph] Implementation Plan ù An ordered, numbered list of tasks with per-task risk level, estimated file scope, and checkpoint flags.
[List Paragraph] Activity Log ù A timestamped, append-only log of every action taken by the framework during the workflow.
[Heading 2] Code Artifacts
[List Paragraph] Generated Project Repository ù A complete, runnable project built from scratch, structured according to the plan produced in Phase 1.
[List Paragraph] Test Suite ù Automatically generated tests (written TDD-first) with a documented pass rate. At minimum, one test per implemented task.
[List Paragraph] Docker Compose File (Optional) ù A docker-compose.yml that builds and runs the generated project in an isolated environment.
[Heading 2] Summary Artifacts
[List Paragraph] Workflow Summary Report ù A final report showing: tasks completed, tasks skipped, files generated, tests passed/failed, and total API calls made.
[List Paragraph] Security Audit Report (if FR-14 implemented) ù A summary of flagged vulnerabilities and the frameworkÆs resolution of each.
[Heading 1] 9. Hackathon Phases
[Normal] The hackathon is conducted in two sequential phases. Phase 1 is an elimination round. Only teams that pass Phase 1 proceed to Phase 2.
[Heading 2] Phase 1 ù Design Presentation (Elimination Round)
[Normal] In this phase, teams present their proposed architecture and approach. No implementation is required or expected. Judges evaluate the quality of thinking, not working code.
[Normal] Deliverable: A design document or presentation (PPT or PDF) covering the following:
[List Paragraph] Agent Architecture ù Which agents will the framework use? What is each agent's responsibility?
[List Paragraph] Workflow Design ù How do agents hand off work to each other? Where are the human-in-the-loop checkpoints?
[List Paragraph] Failure Strategy ù How will the framework detect, handle, and recover from failures?
[List Paragraph] Tech Stack Justification ù Which LLM(s) and orchestration framework(s) will be used, and why?
[List Paragraph] Risk Assessment ù What are the hardest parts of the implementation, and how do you plan to mitigate them?
[Normal] Outcome: A subset of teams is selected to advance to Phase 2. Judges will provide brief feedback to eliminated teams.
[Heading 2] Phase 2 ù Final Implementation & Demo
[Normal] Selected teams implement their framework over the remaining hackathon period. At the end, each team delivers a live demo in front of the judges.
[Normal] The demo must include:
[List Paragraph] A live run of the framework receiving a project specification from one of the complexity tiers (revealed at the start of Phase 2).
[List Paragraph] The framework autonomously clarifying requirements, generating a plan, producing code, running tests, and delivering a summary report ù all in real time.
[List Paragraph] At least one deliberate failure scenario handled gracefully by the recovery module.
[List Paragraph] A walkthrough of the generated project code demonstrating it is functional and test-passing.
[Heading 1] 10. Evaluation Criteria
[Normal] Phase 1 and Phase 2 are evaluated separately. Phase 1 evaluates design quality only. Phase 2 evaluates the working system.
[Heading 2] Phase 1 ù Design Evaluation (Pass / Fail)
[Heading 2] Phase 2 ù Final Evaluation (100 Points)
[Heading 1] 11. Constraints & Guardrails
[Heading 2] 11.1  Technology Constraints
[List Paragraph] Teams may use any LLM API of their choice (e.g., OpenAI, Anthropic, Google, Mistral, or open-source models). Teams must use their own API keys.
[List Paragraph] The framework must generate code for a real, runnable project ù not a simulated or stub implementation.
[List Paragraph] Docker is optional. If implemented, the generated project must pass a docker compose up health check.
[List Paragraph] The backend of any generated project must be in Python. The frontend, if required by the project tier, must be in Angular or React.
[Heading 2] 11.2  Safety Constraints
[List Paragraph] The framework must never execute shell commands that delete files outside the project working directory.
[List Paragraph] API keys must not be hard-coded in source files. Any submitted code containing hard-coded API keys will be disqualified.
[List Paragraph] Teams must not share or reuse any pre-written code as the ôgeneratedö output. All project code must be genuinely produced by the framework during the demo.
[Heading 2] 11.3  Ethical Constraints
[List Paragraph] Teams must not use the framework to generate code that contains malicious logic, even as a demonstration of capabilities.
[List Paragraph] All AI-generated code submitted as part of the deliverable must be reviewed and understood by at least one team member.
[Heading 1] 12. Provided Resources
[Normal] Teams are responsible for sourcing their own LLM API access, development environment, and any additional tooling or libraries they choose to use.
[Heading 1] 13. Submission Requirements
[Heading 2] Phase 1 Submission
[Normal] Submit the following before the Phase 1 presentation deadline:
[List Paragraph] Design document or presentation (PPT or PDF) covering the five areas listed in Section 9.
[List Paragraph] One-paragraph team introduction ù names, backgrounds, and why this problem interests the team.
[Heading 2] Phase 2 Submission
[Normal] Selected teams must submit all of the following before the Phase 2 demo:
[List Paragraph] GitHub repository link ù public or shared with judges ù containing all framework source code.
[List Paragraph] README.md ù with setup instructions, config file documentation, and a brief description of the frameworkÆs architecture.
[List Paragraph] Architecture diagram ù updated from Phase 1 to reflect the actual implementation, showing agent interactions and data flow.
[List Paragraph] Reflection document (1 page max) ù key design decisions made, what the team would do differently with more time, and an honest assessment of current limitations.
[Normal] Submissions missing any required item will be considered incomplete and may be disqualified from judging.
[Heading 1] 14. Notes for Judges
[Normal] This section is intended to help judges apply the evaluation criteria consistently.
[List Paragraph] A team with a reliable Tier 1û2 framework should score higher than a team that attempts Tier 4û5 but produces brittle, frequently failing output.
[List Paragraph] The TDD-first criterion is non-negotiable. A framework that writes tests after the code, or skips tests entirely, cannot score full marks in the Verification category regardless of other quality.
[List Paragraph] Phase 2 demos must use the project tier revealed at the start of Phase 2. Teams may not substitute an easier tier or use a pre-prepared project.
[List Paragraph] Partial credit is available for incomplete Extended features if the team clearly articulates design intent and remaining work.
[List Paragraph] A framework that detects failures, recovers gracefully, and communicates clearly is more impressive than one that only works on the happy path.
[Heading 1] Appendix A: Glossary
[Heading 1] Appendix B: Suggested Agent Roles
[Normal] Teams are free to structure their framework however they choose. The following is one illustrative decomposition, not a required architecture:
[List Paragraph] Intake Agent ù Receives raw specification, identifies ambiguity, generates clarifying questions, and produces the structured specification.
[List Paragraph] Architect Agent ù Designs project structure, directory layout, data models, and API contracts from scratch.
[List Paragraph] Planner Agent ù Converts the structured specification and architecture into an ordered atomic task list.
[List Paragraph] QA Agent (TDD-First) ù Writes failing test cases before any production code is generated for each task.
[List Paragraph] Coder Agent ù Invokes the chosen LLM to generate production code that satisfies the failing tests.
[List Paragraph] Security Auditor Agent ù Performs a pre-commit audit, flagging injection vulnerabilities, auth flaws, and logic errors.
[List Paragraph] Recovery Agent ù Diagnoses failures and decides between retry, guidance request, or escalation.
